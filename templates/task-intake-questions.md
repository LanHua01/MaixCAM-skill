# Task Intake Questions

先读取用户上传的题面、图片、接线图、协议和模型；只追问会改变方案的缺失项。采用核心后分支：先确认核心配置和模块存在性，再按实际存在模块提问。第一阶段未完成时，不生成最终工程、固定协议或 MaixPy 代码。

## 第一阶段：首次配置门禁

### 核心配置

1. 题型、完整任务目标、评分指标、题面或样例是什么？
2. 识别对象、输出对象、精度、时限、误检/漏检代价和失效动作是什么？
3. MaixCAM Pro 的系统/固件版本、相机型号、镜头、分辨率、安装高度和方向是什么？
4. 场地光照、背景、目标颜色/尺寸、距离和安装方式是否固定？是否限制联网、PC、相机数量、补光、模型、无线通信或外部传感器？
5. 是否已有目标 FPS、最大端到端时延、连续运行时长或允许的性能降级？
6. 比赛还剩多少时间，基础/提高/发挥目前分别完成到哪一步，何时必须停止扩展并冻结接口？若未知，先记录“待确认”，不填写绝对工期。

### 模块存在性

先只确认“有 / 无 / 待确认”，不在本轮追问细节：

1. 是否需要向下位机、地面站或其他设备外发数据？
2. 是否存在执行机构、急停、限幅、激光/飞行等安全联锁？
3. 是否需要独立屏幕 UI 或现场调参输入？
4. 是否需要复杂类别/外观语义识别，且传统颜色、几何或编码标记可能不足？

### 条件分支（仅在对应模块存在时）

- **通信/下位机**：通信对象、引脚、电平、供电、端口、波特率、已有协议；是否需要心跳、ACK、序号、重发、查询响应或超时检测及其动作。
- **执行机构/安全**：执行对象、已有保护逻辑、急停、限幅、低速模式和失效动作。
- **独立 UI/调参**：屏幕像素/方向、需显示的画面/状态/错误/性能、触摸/按键/串口输入和无 UI 替代观测方式。
- **YOLO/模型**：MaixPy 版本、已有图片/标注/`.mud`、类别表、数据许可和模型资源；无数据时进入采集与训练计划。

### MaixVision 调试补充（仅在涉及 IDE、部署或运行异常时）

1. MaixVision、固件/MaixPy、板卡型号和主机系统版本分别是什么？
2. 通过何种方式连接设备；错误出现于连接、在线运行、完整项目运行、预览、上传、打包还是安装？
3. 请提供完整控制台日志、最小复现步骤，以及单文件还是包含模型/资源的项目。
4. 资源是否已部署到设备侧；电脑端的红线/无补全是否与板端实际运行错误同时出现？
5. 是否要求交付可直接打开的完整工程文件夹？工程名称、入口、模型/资源文件和在线运行或打包方式是否已有约定？

## 第二阶段：方案确认

核心配置和相关条件分支完成后，先使用 `scoring-roadmap.md` 建立基础/提高/发挥路线，再按 `solution-options.md` 输出必要的方案比较，等待用户选择后再问：

1. 最终采用哪套识别路线及其降级路线？
2. 最终协议、字段、单位、频率、ACK/心跳和异常策略是什么？
3. 是否最终需要屏幕 UI？若需要，必须显示哪些画面、数值、状态、错误、性能和调参入口？
4. 心跳、ACK、序号、重发、超时和丢失策略是否需要；其周期、触发条件和下位机动作是什么？
5. 各模块的输入/输出、独立测试脚本和验收条件是什么？
6. 若使用 YOLO，是否已有图片、标注或 `.mud` 模型？没有时是否先执行数据采集与训练计划？
