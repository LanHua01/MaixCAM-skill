## 工程规模选择

先选择规模，再列出目录；不以文件数量衡量工程化。紧凑和完整工程都必须可在 MaixVision 打开项目文件夹后运行完整项目。

| 规模 | 适用条件 | 交付边界 |
| --- | --- | --- |
| 最小验证 | 单一相机、显示、UART 或 API 问题；不构成正式项目 | 仅交付自包含测试、预期现象和失败排查，不生成完整目录或最终协议。 |
| 紧凑工程 | 单视觉任务、无下位机闭环、无独立 UI/模型，或这些能力尚未确认 | 默认唯一 `main.py`；仅在参数独立调节/多处复用、拆分能降低复杂度或存在独立风险时增加文件；首启可观察。 |
| 完整闭环工程 | 多任务、下位机、UI、模型、执行机构、安全闭环或多个独立风险实际存在 | 使用对应的通信、显示、任务、测试、模型和资源模块；只保留实际使用的部分。 |

工程目录必须服从 `scoring-roadmap.md`：每个文件映射到题面评分项或独立风险；提高/发挥模块必须有关闭开关或回退点，且基础得分路径可独立运行。

## 完整闭环工程的模块参考

```text
<project-name>/
  main.py                  # 唯一运行入口
  config.py
  communication.py
  display_status.py
  tasks/
    __init__.py
    task_<name>.py
  tests/
    camera_test.py
    startup_contract_test.py
    resource_manifest_test.py
    vision_test.py
    performance_test.py
    uart_test.py
    protocol_test.py
    actuator_safe_test.py
  models/                  # 仅放用户已提供或已获授权的模型
  assets/                  # 项目运行必需的图片、配置等资源
  app.yaml                 # 仅在用户要求打包/安装且规范已确认时加入
```

交付时列出实际生成的全部文件及其任务理由，不留“请自行新建”或未说明的外部依赖。用户在 MaixVision 打开 `<project-name>/` 后运行完整项目；多文件、模型或资源项目不应使用“运行当前文件”。紧凑工程可以把首启显示和单视觉任务放在 `main.py`，无需因此创建 `communication.py`、独立显示模块、模型目录或未执行的测试。

- `main`：初始化、调度、性能统计和异常兜底。
- `config`：相机、阈值、ROI、模型、端口、频率、UI 和调试开关。
- `tasks`：输入图像/时间，输出可解释状态；不直接写串口或显示。
- `communication`：字节流、分包、协议和连接快照；不写视觉算法。
- `display`：只读状态，显示 UI 和调参事件；不篡改识别结果。
- `tests`：每个高风险模块的最小验证，不依赖完整闭环。

## 紧凑工程默认目录

```text
<project-name>/
  main.py                  # 唯一入口：相机、单一视觉任务与必要首启观测
```

仅在下列条件满足时增加对应文件：

| 可选文件 | 增加条件 | 不增加时的处理 |
| --- | --- | --- |
| `config.py` | 参数需要独立现场调节、记录，或被多个模块复用 | 将少量固定参数保留在 `main.py` 的配置区。 |
| `tasks/task_<name>.py` | 拆分后能明显降低 `main.py` 的任务复杂度 | 单一视觉流程留在 `main.py`。 |
| `tests/<risk>_test.py` | 某硬件、算法或接口存在需要独立复现的风险 | 用 `main.py` 的可观察首启完成基础验证。 |

## 工程级注释设计

生成代码前，为下列位置规划中文关键注释；注释说明“为什么、契约或风险”，不逐行解释 Python 语法，也不翻译 API、标识符或协议字段。

| 位置 | 必须说明的关键内容 | 不应写成 |
| --- | --- | --- |
| 文件/模块入口 | 职责、相机/屏幕/UART/模型等硬件或资源假设、缺失时的可观察降级 | 每条 `import` 的中文翻译 |
| `config.py` | 参数单位、已确认的范围或来源、对精度/帧率/安全的影响 | 未核验数值的“固定最佳值” |
| `tasks` 公共接口 | 输入/输出、坐标/单位、有效性、错误或目标丢失语义 | 给每个局部变量重复命名含义 |
| 状态机与缓存 | 候选、确认、缓存、丢失的转换条件和防抖目的 | 仅复述 `if` 条件 |
| 通信/执行与主循环 | 发布限频、超时、限幅、急停、双缓冲或调度取舍的原因 | 在未确认前把协议字段写死 |
| 独立测试 | 实际硬件假设、预期画面/收发结果和关键失败条件 | 为十余行最小测试添加逐行注释 |

完整工程应在每个与题目相关的表项处具备对应注释；不相关的硬件、模型或状态无需为凑覆盖率虚构注释。
