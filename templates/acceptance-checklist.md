# Acceptance Checklist

按实际题目裁剪。先把题面逐条映射到得分证据，再执行独立硬件与视觉 → 通信 → 低速闭环 → 整题动作 → 全场景与公开发布检查。

## 题面条款与得分验收矩阵

| 题面条款 | 分值层级/分值 | 实现模块 | 验证动作 | 实测证据 | 得分影响 | 状态 |
| --- | --- | --- | --- | --- | --- | --- |
| 待填 | 基础/提高/发挥 | 待填 | 待填 | 待填 | 待填 | 未开始/待实测/通过/失败 |

- [ ] 基础、提高、发挥分别验收，没有用“整体完成”代替逐条结果。
- [ ] 视觉识别正确后，整题的运动、动作顺序、时限、停留、显示/上报和安全规则仍逐项通过。
- [ ] 每个提高/发挥功能有关闭开关或回退点，并证明关闭或失败时不破坏基础得分路径。
- [ ] 失败项已记录会损失的分值/层级及对其他条款的连带影响。
- [ ] 未在目标板或真实场地验证的条目标为“待实测”，没有计为通过。
- [ ] 已记录比赛剩余时间、停止扩展条件、稳定性预留窗口和接口冻结时刻。

## 相机、显示与视觉

- [ ] 焦距、曝光、白平衡、分辨率和帧率已在赛场环境验证。
- [ ] 屏幕像素、方向、触摸映射和关键状态显示正确。
- [ ] 已确认是否需要 UI；需要时，画面、数值、状态、错误、性能信息和调参入口满足约定；不需要时有替代观测方式。
- [ ] 候选、确认、缓存、丢失状态及失效动作明确；确认/丢失阈值与目标选择规则已验证。
- [ ] ROI、阈值、尺寸、几何或标定误差已用题目指标验证。
- [ ] 单帧异常不会直接触发危险动作。

## 模型

- [ ] `.mud` 文件、类别顺序、输入尺寸和图像格式一致。
- [ ] 数据来源和标注许可已确认；训练/验证/赛场测试没有混淆。
- [ ] 置信度、IOU、ROI、尺寸与时序过滤已在现场测试。
- [ ] `dual_buff=True` 的一帧结果延迟已被允许、补偿，或已关闭。
- [ ] 模型加载失败、低置信度或目标丢失时有安全降级。

## 通信与闭环

- [ ] GND 共地、RX/TX 交叉、逻辑电平、端口和波特率正确。
- [ ] UART0 使用时已处理启动日志和 A16 上电风险；否则使用 UART1。
- [ ] 解析器支持粘包、拆包、超时、异常字节和日志限流。
- [ ] 协议、字段、单位、频率、ACK/心跳由用户或题面确认。
- [ ] 心跳、ACK、序号、重发、查询响应、超时和丢失策略逐项确认；未选择的机制没有被默认实现。
- [ ] 已有工程的已确认引脚、协议、波特率和安全联锁未被无故替换；固定测试包与下位机解析已先行验证。
- [ ] 单轴/单机构低速、方向、限幅、急停和目标 lost 策略通过。

## 工程化与公开发布

- [ ] 主循环、任务、通信、显示、配置和独立测试职责清晰，无无意义重复逻辑。
- [ ] 文件必要性：已选择最小验证、紧凑工程或完整闭环工程；每个生成的文件/目录都有题目职责理由，无通信/UI/模型/独立风险时未创建空模块、无效测试或无用资源目录。
- [ ] 首次问询先完成核心配置和模块存在性；未存在的通信、执行、独立 UI、调参输入或模型分支未被追问细节，也未进入最终工程。
- [ ] 方案先满足题面、时限、安全、精度和性能预算，再在合格路线中选择最少依赖、最少标定和最少接口的实现；YOLO 未因存在公开案例而被无故引入。
- [ ] 相机、算法、串口、协议和执行机构均可单独验证。
- [ ] 工程级关键注释覆盖相关的模块职责/硬件假设、配置单位与影响、公共任务接口的输入输出和坐标/失效语义、状态转换、非直观时序及安全降级；无关项不为凑数量虚构注释。
- [ ] 生成的工程代码注释为中文，且说明原因、契约或风险，而非逐行复述代码；未改写 Python 或 MaixPy API 名称。最小测试只保留硬件假设、预期现象和关键失败条件。
- [ ] 正向代码示例已对照 MaixPy 官方文档；未使用 OpenMV/K210/PC 专属 API。
- [ ] 输出前已检查 MaixVision/板端运行时边界、Python 语法、UART 字段数和版本敏感 API 假设。
- [ ] 涉及 MaixVision 时，已分别验证设备连接、最小相机显示、完整项目资源、在线运行与打包/安装后的行为。
- [ ] 已记录源码快照、工程/资源清单、运行/部署方式、设备运行身份和分阶段日志；每个结论可关联到同一次代码、部署与板端运行。
- [ ] 跨模块接口的参数、类型/单位/范围、返回字段、错误状态和配置键已记录并通过独立契约测试；接口变更未只修改调用方或实现方之一。
- [ ] 多文件/资源/打包工程已核对实际入口、导入和部署清单；未把编辑器可见文件、Git 工作区或旧包误当作设备实际运行版本。
- [ ] 已按题目建立场景回归：至少覆盖远近、光照/反光、遮挡/空场、背景或目标切换和资源缺失；记录配置、期望、实际与闭环安全影响。
- [ ] 已交付唯一 `main.py` 入口和完整工程目录；在 MaixVision 打开文件夹并运行完整项目后可进入可观察的调试状态。
- [ ] 多文件、模型和资源均随工程提供并使用相对路径；缺失模型/资源时显示安全降级，而非伪造最终可运行结果。
- [ ] 已在目标 MaixCAM/MaixCAM Pro 记录相机、视觉、模型、UI/串口、闭环和连续运行的性能；未实测项明确标为待实测。
- [ ] 性能问题已附复现条件、日志、瓶颈判断、优化代价和精度/时序/安全回归结果。
- [ ] 同一故障连续两次修改失败后已停止补丁式修改，并审查源码、工作区、入口、配置、资源哈希、部署包、板端身份和日志的一致性。
- [ ] 每轮保留状态卡：冻结接口、本轮范围、最后板端通过版本、当前问题、修改结果、回退点和待实测项。
- [ ] 发布文本不含姓名、账号、邮箱、手机号、本机路径、令牌、密码、私钥、内网地址或未经授权的数据集。
- [ ] 外部链接可公开访问，模型、图片和标注的来源及许可已确认。
- [ ] 条件自动检索触发时，研究卡清楚区分官方事实、公开经验和待确认项；搜索结果未覆盖用户已确认的题面、接线、协议或安全约束。
- [ ] 研究卡仅在外部资料会核验事实或改变候选路线时出现，且最多保留一条一级事实和两条必要二级经验。
