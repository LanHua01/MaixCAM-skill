# MaixVision Debugging Guide

当用户在 MaixVision 中遇到连接、运行、预览、上传、补全、打包或日志错误时读取本文件。MaixVision 是开发、部署和调试环境；板端代码的运行时仍是 MaixPy。

核心原则：不要把“编辑器文件存在”或“某次能运行”当作部署成功。对多文件、资源或打包工程，必须能追溯 `源码快照 → 工程清单 → 运行/部署方式 → 设备运行身份 → 资源与接口 → 分阶段现象`。

## 先建立可复现证据链

使用 `templates/maixvision-debug-evidence.md`，先建立本次运行记录。至少保留：

- 源码快照：当前分支/提交号；没有 Git 时使用用户确认的版本标签、修改时间和关键文件清单。不要把本机绝对路径写入公开记录。
- 工程清单：唯一 `main.py`、本地模块、模型、资源、配置和实际导入关系；标明本次是“运行当前文件”“运行完整项目”还是打包/安装。
- 设备运行身份：板卡、固件/MaixPy、MaixVision、连接方式、运行时间、部署方式和可公开的构建/包标识。未取得的信息标为待确认，不得编造版本或哈希。
- 运行证据：从启动、配置、资源、相机、视觉、通信到安全降级/异常的阶段日志；每条日志至少能关联阶段、模块、运行身份、配置/输入摘要和错误码/现象。

这不是要求固定日志格式、Git、打包方式或设备路径，而是要求每个结论可对应到同一次源码、部署和设备运行。

## 先收集最小证据

先索取而不是猜测：MaixVision 版本、主机系统、板卡型号、固件/MaixPy 版本、连接方式、运行方式（当前文件或完整项目）、完整错误/控制台日志、最小复现步骤，以及是否涉及模型或资源文件；多文件问题还要索取源码快照、工程清单、部署/包标识和设备侧实际运行身份。

如果用户只能提供现象，先让其分别验证“设备可连接”“最小相机显示可运行”“资源可在设备侧读取”；不要一开始把问题归因于视觉算法。

## 诊断分流

| 现象 | 优先检查 | 下一步 |
| --- | --- | --- |
| 找不到或无法连接设备 | 供电、数据线、连接方式、设备识别与版本信息 | 换用最小连接方式并保留完整报错；确认后再处理驱动、网络或固件问题。 |
| 在线运行失败 | 当前文件是否自足、板端日志、import、语法、设备侧资源 | 先运行最小 `camera + display`，再逐项恢复模块。 |
| 多文件项目运行失败 | 是否误用“运行当前文件”、入口文件、依赖和资源是否随项目发送 | 使用完整项目运行；检查入口和相对资源路径，不以电脑端文件是否存在为依据。 |
| 预览黑屏、停帧或画面不对 | 相机初始化、`disp.show(img)`、显示方向、分辨率、帧率与异常日志 | 先只显示相机帧和 FPS；再检查曝光、算法和叠加绘制。 |
| 编辑器红线或无补全 | 电脑端 Python/MaixPy 提示环境、解释器配置与版本匹配 | 明确这可能只影响编辑体验；仍必须在目标板上实际运行验证。 |
| 电脑/编辑器、Git 工作区、包或板端行为不一致 | 源码快照、工程清单、运行方式、部署/包标识和设备运行身份是否来自同一次记录 | 先补齐证据链，再比较差异；不以“我刚保存过”代替部署验证。 |
| 模块导入、返回字段或配置改变后异常 | 模块接口契约：参数、类型/单位/范围、返回结构、错误和配置键 | 先做独立契约测试，再恢复完整视觉链路；不要靠调用方猜测兼容性。 |
| 找不到图片、模型或配置 | 电脑文件系统与设备文件系统是否混淆、资源是否上传/随项目部署 | 在设备侧列出或验证资源；使用项目内相对路径，不写入电脑绝对路径。 |
| 工程清理、打包或安装后出现旧模块/旧资源 | 实际入口、工程清单、资源清单、打包/安装标识和设备侧版本日志 | 删除或排除未使用资源前先确认依赖；在线完整项目成功后，再单独验证包/安装物。 |
| 单一距离/光照可用，赛场失效 | 场景矩阵是否覆盖远近、过曝/弱光、反光、遮挡、空场和目标切换 | 将失败场景、参数和结果写入回归记录；不要只继续微调单一阈值。 |
| 难以判断是算法、资源还是部署错误 | 日志是否有阶段、模块、运行身份、输入/配置摘要和关键中间结果 | 先补充不含敏感信息的结构化观测，再决定修改算法或重新部署。 |
| 打包/安装后行为不同 | 打包入口、应用标识、模型和资源是否被包含、固件兼容性 | 先回到在线项目运行验证，再用最小项目打包；保留打包日志。 |

## 推荐调试顺序

1. 建立调试证据记录：保存完整错误、版本、最小复现、源码快照、工程清单和本次运行方式；不要只截取最后一行报错。
2. 验证设备连接与板端日志通道，确认设备运行身份，再运行最小相机/显示程序。
3. 若使用多个 Python 文件、模型或图片，切换到完整项目运行，按工程清单核对入口、导入与设备侧资源。
4. 对模块边界不确定处先核对接口契约和独立测试：参数/单位/范围、返回结构、错误和配置键必须由实现与调用方共同确认。
5. 通过 `disp.show(img)` 观察真实图像和叠加状态；预览异常时先移除算法和通信，并保留关键中间结果。
6. 依次恢复视觉、模型、串口和闭环；每次只恢复一个模块，记录运行阶段、输入条件和结果。
7. 在线完整项目正确后，再验证打包/安装与离线启动；确认版本、资源、配置和行为与同一次证据链一致。

## 可直接打开的工程检查

在交付给 MaixVision 前，检查项目根目录：

- 存在唯一 `main.py`，并且其本地导入、配置、模型和资源都包含在工程目录内。
- 多文件项目使用“打开文件夹/项目 + 运行完整项目”；不要将“运行当前文件”当作完整工程部署。
- 所有资源路径相对项目目录或设备部署目录解析，不引用电脑绝对路径。
- 第一次启动先显示画面、FPS、状态或错误；模型、资源或硬件缺失时进入可见的安全降级，而不是静默失败。
- 在线项目运行成功后再尝试打包/安装；若两者行为不同，分别保存日志并从入口、资源清单和版本差异排查。
- 首启状态需能显示或输出启动阶段、模块、运行身份、配置/资源是否就绪、输入摘要、关键中间结果或安全降级原因；不要只输出“运行失败”。
- 识别任务至少按题目相关场景建立回归矩阵：近/远、强光/弱光或反光、遮挡/空场、背景变化、目标切换和资源缺失。每个场景记录输入条件、配置、期望、实际和是否影响闭环安全。

## 特别边界

- MaixVision 的代码补全依赖电脑端的 Python 环境；补全异常不自动证明板端 MaixPy 不能运行。
- 单文件运行通常只适合自包含测试；需要多个模块、模型或其他资源时，应以完整项目运行和设备侧资源检查为准。
- 实时预览依赖板端实际调用显示链路；没有画面时先检查最小相机与显示代码，而不是先调模型阈值。
- 不在公开文档、日志或示例中写入个人设备名、电脑绝对路径、账号、密码、令牌或未授权模型。
- 不将 Git、哈希、固定包名、固定设备路径或固定日志字段当作唯一实现；它们是可替换的运行身份载体。关键是同一次代码、工程、部署和设备运行可被关联和复现。
