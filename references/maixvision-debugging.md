# MaixVision Debugging Guide

当用户在 MaixVision 中遇到连接、运行、预览、上传、补全、打包或日志错误时读取本文件。MaixVision 是开发、部署和调试环境；板端代码的运行时仍是 MaixPy。

## 先收集最小证据

先索取而不是猜测：MaixVision 版本、主机系统、板卡型号、固件/MaixPy 版本、连接方式、运行方式（当前文件或完整项目）、完整错误/控制台日志、最小复现步骤，以及是否涉及模型或资源文件。

如果用户只能提供现象，先让其分别验证“设备可连接”“最小相机显示可运行”“资源可在设备侧读取”；不要一开始把问题归因于视觉算法。

## 诊断分流

| 现象 | 优先检查 | 下一步 |
| --- | --- | --- |
| 找不到或无法连接设备 | 供电、数据线、连接方式、设备识别与版本信息 | 换用最小连接方式并保留完整报错；确认后再处理驱动、网络或固件问题。 |
| 在线运行失败 | 当前文件是否自足、板端日志、import、语法、设备侧资源 | 先运行最小 `camera + display`，再逐项恢复模块。 |
| 多文件项目运行失败 | 是否误用“运行当前文件”、入口文件、依赖和资源是否随项目发送 | 使用完整项目运行；检查入口和相对资源路径，不以电脑端文件是否存在为依据。 |
| 预览黑屏、停帧或画面不对 | 相机初始化、`disp.show(img)`、显示方向、分辨率、帧率与异常日志 | 先只显示相机帧和 FPS；再检查曝光、算法和叠加绘制。 |
| 编辑器红线或无补全 | 电脑端 Python/MaixPy 提示环境、解释器配置与版本匹配 | 明确这可能只影响编辑体验；仍必须在目标板上实际运行验证。 |
| 找不到图片、模型或配置 | 电脑文件系统与设备文件系统是否混淆、资源是否上传/随项目部署 | 在设备侧列出或验证资源；使用项目内相对路径，不写入电脑绝对路径。 |
| 打包/安装后行为不同 | 打包入口、应用标识、模型和资源是否被包含、固件兼容性 | 先回到在线项目运行验证，再用最小项目打包；保留打包日志。 |

## 推荐调试顺序

1. 保存完整错误、版本和最小复现；不要只截取最后一行报错。
2. 验证设备连接与板端日志通道，再运行最小相机/显示程序。
3. 若使用多个 Python 文件、模型或图片，切换到完整项目运行，核对设备侧资源。
4. 通过 `disp.show(img)` 观察真实图像和叠加状态；预览异常时先移除算法和通信。
5. 依次恢复视觉、模型、串口和闭环；每次只恢复一个模块。
6. 在线运行正确后，再验证打包/安装与离线启动；确认版本、资源和配置一致。

## 可直接打开的工程检查

在交付给 MaixVision 前，检查项目根目录：

- 存在唯一 `main.py`，并且其本地导入、配置、模型和资源都包含在工程目录内。
- 多文件项目使用“打开文件夹/项目 + 运行完整项目”；不要将“运行当前文件”当作完整工程部署。
- 所有资源路径相对项目目录或设备部署目录解析，不引用电脑绝对路径。
- 第一次启动先显示画面、FPS、状态或错误；模型、资源或硬件缺失时进入可见的安全降级，而不是静默失败。
- 在线项目运行成功后再尝试打包/安装；若两者行为不同，分别保存日志并从入口、资源清单和版本差异排查。

## 特别边界

- MaixVision 的代码补全依赖电脑端的 Python 环境；补全异常不自动证明板端 MaixPy 不能运行。
- 单文件运行通常只适合自包含测试；需要多个模块、模型或其他资源时，应以完整项目运行和设备侧资源检查为准。
- 实时预览依赖板端实际调用显示链路；没有画面时先检查最小相机与显示代码，而不是先调模型阈值。
- 不在公开文档、日志或示例中写入个人设备名、电脑绝对路径、账号、密码、令牌或未授权模型。
