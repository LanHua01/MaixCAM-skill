# Existing Project and Integration Guide

在用户提供已有 MaixPy 工程、下位机代码、接线表、协议或 MaixVision 工程时读取本文件。目标是接管真实运行链路、一次修一个有证据的根因，而不是另起一套平行工程或连续猜测性打补丁。

## 固定故障接管顺序

```text
确认板端实际版本
→ 阅读入口与调用链
→ 最小复现
→ 分层定位
→ 根因与证据
→ 一次只修一个根因
→ 板端复测
```

没有完成前一步证据时不得跳到大范围重写。源码看起来正确不等于板端正在运行该版本；电脑端模拟通过也不等于 MaixPy 严格参数、资源和性能已经通过。

## 先建立现状清单

- 先阅读入口、配置、视觉任务、通信和下位机解析代码，再提出修改。
- 标记已验证的端口、引脚、电平、波特率、字段、坐标系、限幅和急停逻辑；未经用户确认不得替换。
- 明确本次是修复、调参、增量功能还是架构重做。仅当改动影响高影响决策时，重新给出方案矩阵。
- 优先最小补丁、独立测试和可回退配置；不要为局部问题引入另一条并行主循环或重复协议。

使用下表持续记录，而不是每轮重新猜测：

| 项目 | 当前证据 |
| --- | --- |
| 源码快照/版本标识 | 待记录 |
| 唯一入口与调用链 | 待记录 |
| 配置和资源清单 | 待记录 |
| 接口契约 | 引脚、协议、单位、范围、错误状态，待记录 |
| 运行/部署方式 | 当前文件/完整项目/打包安装，待记录 |
| 板端身份 | 启动日志、版本、构建或资源指纹，待记录 |
| 最后板端通过基线 | 待记录 |
| 同一问题失败修改次数 | 0 |

可先运行 `scripts/inspect_maix_project.py`、`scripts/check_runtime_fingerprint.py`、`scripts/check_maixpy_api_usage.py` 和 `scripts/estimate_frame_cost.py` 生成只读静态证据；这些工具不会自动访问设备，板端状态仍需用户提供日志或实测。

## 两次失败后的全链路审查

同一现象连续两次修改仍未解决时，停止继续调阈值、改参数或叠加兼容分支，按顺序核对：

1. 源码快照和实际编辑工作区是否一致。
2. 唯一入口、导入链与配置读取是否指向预期文件。
3. 模型、图片、配置等资源相对路径及 SHA-256 是否一致。
4. 当前运行的是单文件、完整项目还是打包安装产物。
5. 部署包/清单与板端启动身份是否对应。
6. 分阶段日志是否证明相机、资源、算法、通信和执行运行到预期步骤。
7. 恢复最后板端通过基线后，原问题是否仍可最小复现。

审查完成前不提出第三个猜测性补丁。

## MaixVision、MaixPy 与下位机边界

- MaixVision：工程创建、部署、下载、日志和调试界面。
- MaixPy：板端采集、识别、状态处理、显示和通信 API。
- 下位机：运动控制、执行机构状态机、时序敏感动作和安全联锁。

不要把 MaixVision 的菜单、部署或调试动作当作板端 Python 调用；不要让视觉任务层直接驱动危险执行机构。

## 稳定目标状态

统一把检测结果表示为可解释状态，而不是把每一帧候选直接发送：

```text
candidate -> confirmed -> cached -> lost
```

- `candidate`：当前帧候选，尚不足以控制或上报关键动作。
- `confirmed`：满足连续帧、几何/置信度和目标选择规则的稳定主目标。
- `cached`：短时未见目标但仍在允许保持窗口内；必须带原始时间戳。
- `lost`：超过保持窗口、低置信度或检测链路异常；按确认过的策略停止、保持、搜索或降级。

将确认帧数、丢失帧数/时间、ROI、面积/置信度阈值、目标选择规则和离群阈值放入配置。多目标时只使用一条跨帧一致的选择规则，例如最大面积、最高置信度、最近中心或最接近上一已确认目标。

## UART 联调顺序

1. 用用户确认的固定测试包或串口助手验证物理接线与下位机解析；不要凭空指定帧头、频率或心跳。
2. 在不接控制的条件下验证相机画面、曝光、坐标方向和时间戳。
3. 运行最小识别，只显示候选与原始状态。
4. 加入确认/缓存/丢失状态，验证目标切换、短暂遮挡和误检。
5. 按最终协议发送稳定状态，在低速、限幅、单执行机构条件下验证闭环。
6. 在赛场光照、背景、供电和动作速度下复测；每轮只改变一个参数。

协议层必须区分“目标未找到”“目标过期”“通信掉线”和“解析失败”；具体字段、ACK、心跳、发送周期和重发策略由题目或用户确认。
