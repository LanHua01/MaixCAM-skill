# MaixCAM Performance Validation

在完整工程目录生成后、闭环验收前读取本文件。性能结论只以目标 MaixCAM 或 MaixCAM Pro 的实测为准；开发电脑、编辑器预览或理论 FPS 不能替代板端结论。

## 先定义合格条件

- 板卡型号、固件/MaixPy 版本、相机/镜头、分辨率、图像格式、目标帧率与场地光照。
- 任务允许的端到端时延、控制周期、连续运行时长、精度和安全动作。
- 视觉路线、模型输入尺寸、`dual_buff` 设置、显示模式、串口字段与发送策略。

若题面未规定 FPS 或延迟，先给出与控制需求对应的候选目标并等待用户确认；不要把开发阶段的瞬时最高 FPS 当作验收标准。

## 速率与时延口径

在同一观测窗口内分别记录，避免用一个“FPS”掩盖系统瓶颈：

| 指标 | 记录方法 | 不能替代什么 |
| --- | --- | --- |
| 相机采集速率 | 完成 `cam.read()` 的帧数 ÷ 观测时间 | 不能代表算法、显示或闭环速度。 |
| 视觉处理/有效检测速率 | 完成任务处理的帧数，以及通过有效性判定的结果数 ÷ 观测时间 | 不能以仅有画面或候选框的帧数冒充有效结果。 |
| 控制发布频率 | 实际经确认、限幅和失效策略后发送给执行侧的结果数 ÷ 观测时间 | 不等于 UART 写入次数，也不等于相机 FPS。 |
| 端到端时延 | 关联同一观测结果的采集、确认、发送及可获得的执行侧反馈时间戳 | 没有可关联时间戳时只能报告分段时延和假设。 |

板端运行/调参 UI 是可选负载。启用时单独记录其刷新频率、绘制耗时和输入处理耗时；若 UI 使控制发布频率或端到端时延超标，先降 UI 刷新率/绘制量或关闭 UI，再重新测量，不能只报告相机 FPS。

## 板端测试顺序

1. **相机基线**：只采集和显示，记录暖机后的稳定 FPS、画面方向、曝光和是否掉帧。
2. **视觉基线**：加入颜色/几何等任务，记录 FPS、单帧处理时间、候选数和误检情况。
3. **模型基线**：单独加入模型，记录输入尺寸、推理时间、模型加载是否稳定和 `dual_buff` 对时序的影响。
4. **I/O 基线**：加入屏幕叠加、调试打印和串口，检查发送频率、缓冲积压、丢包/解析异常和 UI 是否拖慢循环。
5. **完整闭环**：低速、限幅运行，记录从图像时间戳到下位机收到有效命令的端到端时延。
6. **稳定性运行**：在赛场光照、背景和供电条件下连续运行规定时长；记录卡顿、异常、内存增长、模型加载失败、重启和恢复能力。

在每一步只新增一个模块。先保存基线，再比较新模块带来的性能变化。

## 必须记录的证据

- 测试配置、开始/结束时间、样本量、平均/最差 FPS 或帧间隔、端到端延迟。
- 相机、视觉、模型、显示、串口和控制是否启用，以及模型输入和双缓冲设置。
- 控制台/设备日志中的异常、重启、资源加载失败或通信错误。
- 复现条件：场地光照、目标数量、遮挡、线缆/供电和动作速度。

用 `templates/performance-report.md` 输出结果。没有实测数据时，明确列出待测项和最小测试步骤。

## 问题定位与优化选项

| 现象 | 优先定位 | 候选优化 | 代价与回归验证 |
| --- | --- | --- | --- |
| 相机基线 FPS 低或画面卡顿 | 分辨率、帧率设置、曝光、显示链路、供电 | 选择满足精度的更低分辨率/帧率，减少显示刷新，固定测试光照 | 复测目标尺寸、坐标精度和连续运行稳定性。 |
| 传统视觉拖慢循环 | ROI、候选数量、全图扫描、重复颜色/几何计算 | 先裁剪 ROI，先做轻量过滤，降低非关键处理频率 | 复测边缘目标、遮挡、快速运动和误检。 |
| 模型推理过慢或控制滞后 | 模型输入、类别/候选、显示、`dual_buff` 时序 | 缩小输入/ROI，减少非关键叠加，改用传统视觉或融合降级；严格同步闭环关闭双缓冲或补偿时间戳 | 对比精度、漏检和端到端时延；不能只看吞吐 FPS。 |
| 串口/显示导致卡顿 | 每帧打印、绘制量、发送频率、缓冲积压 | 限速日志和 UI，按确认状态或固定周期发送，缩短已确认帧 | 复测下位机解析、状态更新时延和调试可观测性。 |
| 长时间运行崩溃、重启或资源增长 | 异常日志、重复加载模型/资源、临时对象、供电和温升 | 将模型/资源初始化移出主循环，复用对象和缓冲，限制日志/缓存，检查供电与散热 | 执行等长或更长稳定性运行，确认无新错误和性能衰减。 |

不要默认使用未验证的并发、桌面优化库或高风险重构。每次优化只改变一个因素，并保留前一份可运行配置以便回退。
