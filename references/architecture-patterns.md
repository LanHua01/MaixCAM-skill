# Architecture Patterns

MaixCAM Pro 电赛项目以“可替换方案、可独立验证、可安全降级”为目标。分层不是增加文件数量；每层只承担一个稳定职责。

## 方案确认后的最小分层

| 层 | 只负责 | 不负责 |
| --- | --- | --- |
| `main` | 初始化、调度、任务切换、性能与异常兜底 | 识别细节、协议分包、复杂控制分支 |
| `config` | 相机、屏幕、引脚、阈值、ROI、模型、端口、UI、调试参数 | 算法流程与设备 I/O |
| `tasks` | 输入图像/时间，输出可解释 `state` | 直接写串口、显示或执行机构 |
| `communication` | I/O、字节流、分包、协议、连接快照 | 视觉算法和 UI 决策 |
| `display` | 图像叠加、状态栏、错误、调参事件 | 修改识别结论或协议 |
| `tests` | 相机、视觉、模型、串口、协议、执行安全的最小验证 | 替代正式主循环 |

生成工程时，将这些层放入一个可直接打开的项目根目录；`main.py` 是唯一板端入口，所有被其导入的本地模块、模型和资源必须随项目目录存在。不要把测试片段、模型路径或配置留在用户电脑的其他目录。

## 数据流与状态契约

```text
camera.read()
  -> task.process(img, now_ms)
  -> state
  -> communication.encode(state-derived payload)
  -> display.render(img, state, comm_snapshot)
```

`state` 至少包含任务名、目标列表、主目标、`candidate/confirmed/cached/lost`、来源、置信度、时间戳、调试信息和错误。通信层从状态派生用户确认的 payload；显示层只读状态。稳定状态的含义和配置见 `existing-project-integration.md`。

## 方案可替换点

- 视觉：颜色、几何、标定、YOLO 或融合任务通过相同状态契约切换。
- 通信：文本、二进制、JSON、Modbus 或已有协议仅替换通信编码/解码，不侵入任务层。
- UI：屏幕分辨率、方向、状态栏和触摸入口在配置/显示层定义。
- 安全：目标丢失、低置信度、通信离线、模型失败和执行器故障都输出明确状态，由用户确认停止、保持、搜索或降级策略。

## 独立验证顺序

1. `camera_test`：画面、FPS、焦距、曝光、屏幕方向。
2. `vision_test`：只画候选、坐标和状态，不接控制。
3. `model_test`：模型、类别、输入尺寸、延迟和误检。
4. `uart_tx/rx/protocol_test`：先用已确认的测试包验证原始字节、分包、异常流和下位机解析。
5. `actuator_safe_test`：方向、限幅、急停、单轴低速。
6. `performance_test`：在目标板记录相机、视觉、模型、UI、串口和连续运行性能。
7. 闭环联调：从低速和保守模式开始，每轮只改一个参数。

## 反模式

- 在主循环堆叠识别、UI、协议和控制细节。
- 让显示逻辑修改算法状态，或让任务层直接写串口。
- 用未确认的固定帧格式、单位或模型作为项目默认。
- 只有完整闭环演示，没有任何最小测试。
